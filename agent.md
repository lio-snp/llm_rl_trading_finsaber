## 0) First Principles (Non-negotiable)

### P0.1 Experiment-first (Design → Build → Run → Report)
Any meaningful change must start from a **1-page design**. Code is the implementation of the design, not the other way around.

### P0.2 Separation of concerns
- Data lives only in `data/**` (never in git).
- Code lives only in `src/**` and entry scripts in `scripts/**`.
- Config lives only in `configs/**` (YAML).
- Writing lives only in `writing/**`.

### P0.3 No hard-coded paths
No absolute paths. All paths are resolved from project root + YAML configs.

### P0.4 Idempotent & reproducible
Same inputs + same config + same code ⇒ same outputs.
All runs must go through the unified entry:
- `python scripts/run.py <target>`
which snapshots configs and generates run logs.

### P0.5 Falsifiable-by-default (recommended as a MUST)
If a claim can’t survive basic falsification tests (alignment break / time shift / mapping break / zero-channel), it is not a reliable result.
Exception is allowed only for **smoke tests**, explicitly documented.

### P0.6 Figures in English
All plots: title/axes/legend/annotations in English. Main text may be Chinese.

### P0.7 Per-round documentation is mandatory
For **every experiment round** (including reruns), documentation must be updated **twice**:
- **Before run**: update `design.md` + `inputs.md` + `outputs.md` (planned artifacts/protocol).
- **After run**: update `results.md` + `analysis.md` + `decisions.md` + `next.md` (actual evidence/decisions).
If this pre/post record is missing, the round is treated as non-official and cannot be used for conclusions.

## 1) Repository Contract (What goes where)

### 1.1 `data/` (NOT tracked by git)
- `data/raw/`      : read-only raw dumps (append only)
- `data/interim/`  : cleaned/aggregated intermediate
- `data/processed/`: modeling-ready parquet + `_meta/`

`data/processed/_meta/` must include:
- `schema_*.json` (columns/dtypes/keys/frequency/units/missing policy)
- `manifest.csv`  (file hash, generated time, upstream input hash)

### 1.2 `src/` (reusable modules, prefer pure functions)
- `src/data/`      : data loading/feature generation (FINSABER + synthetic)
- `src/env/`       : trading environments and state schema
- `src/drl/`       : TD3/SB3 runners, replay buffer, metrics
- `src/lesr/`      : LESR prompts, candidate generation/parsing
- `src/llm/`       : DeepSeek client + stubs
- `src/pipeline/`  : end-to-end experiment orchestration
- `src/utils/`     : paths, hash, code loading helpers

### 1.3 `configs/` (single source of truth)
- Experiment YAMLs by purpose (current repo pattern):
  - `demo.yaml`, `stage1.yaml`, `stage2.yaml`, `stage1_w05.yaml`, `stage2_w05.yaml`
  - `td3_*` / `td3_matrix/*` / `td3_sac_*` / `td3_quant_ablation/*`
  - `fixed_candidates/*` for fixed revise/intrinsic code
- New experiments should add a new YAML (or YAML family) instead of mutating historical files in place.

### 1.4 `docs/steps/` (the scientific record)
Every step is a folder: `docs/steps/step_xx_<name>/`
Required files (in this order):
- `design.md`   (≤ 1 page)
- `inputs.md`
- `outputs.md`
- `results.md`
- `analysis.md`
- `decisions.md`
- `next.md`
Artifacts:
- `docs/steps/step_xx_*/artifacts/`

Current project also keeps method-level notes in:
- `docs/lesr_share/`
- `docs/diagrams/`

### 1.5 `runs/` (auto-generated, never hand-edit)
Generated by `scripts/run.py`:
- `config.yaml` (snapshot of the used config)
- `metrics.json`, `metrics_table.csv`
- `run_manifest.json`, `artifacts.json`, `hashes.json`
- diagnostic artifacts (e.g., `reward_trace.json`, `td3_g1_g3_diff.json`, `td3_seed_trace.json`)
- walk-forward runs also include `walk_forward_summary.json` and `walk_forward_metrics_table.csv`

### 1.6 `reports/` (human-readable Chinese summaries)
Should reference `docs/steps/*` artifacts and results.

### 1.7 `writing/` (paper)
- Versioned TeX only (`main.tex`, `main_v2.tex`, `main_v3.tex`…)
- Never overwrite older versions
- Build outputs to `writing/paper/build/`

## 2) Standard Workflow (The only way to work)

### Step 0 — Round documentation gate (mandatory every experiment round)
For **every** experiment round (before run + after run), update step docs in `docs/steps/step_xx_<name>/`.

- **Before run (must update first):**
  - `design.md` (goal, hypothesis, protocol, stop condition)
  - `inputs.md` (configs/candidates/data scope for this round)
- **After run (must update immediately):**
  - `outputs.md` (produced files and paths)
  - `results.md` (objective metrics and run ids)
  - `analysis.md` (Claim -> Evidence -> Implication)
  - `decisions.md` (accepted/rejected choices with reasons)
  - `next.md` (next incremental actions)

No run is considered “finished” until the corresponding pre/post docs are updated.

### Step A — Write design first (mandatory)
Create:
`docs/steps/step_xx_<name>/design.md` (≤ 1 page)

Must include:
1) Goal (what you try to establish)
2) Hypothesis (what would be true if successful)
3) Evaluation protocol (metrics, windows, CI)
4) Falsification plan (at least 1–2 tests)
5) Expected outputs (files & schemas)
6) Stop condition / success criteria

Before running, also update:
- `inputs.md` (exact configs, datasets, candidate files, seeds)
- `outputs.md` (expected artifact list and completeness criteria)

### Step A.5 — Literature-first gate for RL mechanism issues (mandatory)
When the problem is RL-behavioral/mechanistic (e.g., collapse, reward shaping, instability):
1) First run a targeted external scan (papers + official code + high-quality technical references).
2) Write a short `literature_scan.md` in the current step folder with:
   - what issue is matched,
   - known solutions,
   - which solutions are adopted/rejected and why.
3) Only then run new experiments.

Exception:
- Skip only for plumbing/smoke runs, and mark in `decisions.md` as `no-literature-scan: smoke-only`.

### Step B — Implement with config-driven code
Rules:
- All tunables in the target config YAML passed to `scripts/run.py` (no magic numbers scattered)
- Pure functions in `src/**`, orchestration in `scripts/**`
- Determinism: explicit seeds, logged

### Step C — Run only via unified entry
- `python scripts/run.py <target>`
Never run ad-hoc commands as “official” runs.
The run must generate a `runs/YYYYMMDD_HHMMSS_<target>/` folder.

### Step D — Write results + analysis + decisions
`results.md` must contain:
- What was produced (file list)
- Basic QC stats: row counts, date ranges, missing rates, class balance, event counts
- Failures/exceptions and how handled

`analysis.md` must follow:
**Claim → Evidence → Implication**
- Claim: one-sentence conclusion
- Evidence: tables/figs/metrics (include negative results)
- Implication: what it means + limitations + next step

`decisions.md` must:
- Record only decisions that matter (window size, mapping, label definition)
- Use objective language: “Based on constraints/evidence, we chose X because…”

`next.md` must:
- Only incremental next actions
- Dependencies + TODOs

Round-close requirement (mandatory):
- Each experiment round must end with updates to `results.md`, `analysis.md`, `decisions.md`, `next.md`.
- Include run ids and artifact paths for every key claim.

## 3) Output Specification (What the agent must deliver)

Every change must produce:
1) A step folder under `docs/steps/`
2) Code changes in `src/**` and/or `scripts/**` (no notebooks as final)
3) Config changes in `configs/**` if parameters change
4) Run logs under `runs/**` for official results
5) Artifacts saved into `docs/steps/.../artifacts/` or `data/**`

**Do not**:
- Commit data
- Hardcode absolute paths
- Overwrite old paper versions
- Present results without a run log + config snapshot

## 4) Evaluation Protocol & Quality Gates (Trading-LESR)

### 4.1 Stage labeling is mandatory for every official run
Each run must declare stage in config (`experiment.phase`) and carry it into `run_manifest`:
- `E0`: audit existing evidence (no new training)
- `E1`: mechanism isolation (controlled exploratory experiments)
- `E2`: solution screening (fixed candidate pool, tunable controls)
- `C0`: confirmatory run (frozen configs/candidates/windows)
- `C1`: paper-level reproduction only (no tuning)

Rules:
- `E*` outputs cannot be used as final paper claims.
- `C*` outputs require frozen hashes and strict reproducibility checks.

### 4.2 Core objective + mechanism objective (must be separated)
Always report both layers:
- **Outcome layer**: `Score = Sharpe + CR` with window-level and aggregate stats.
- **Mechanism layer**: reward-policy-path evidence:
  - reward: `reward_env`, `intrinsic`, `reward_total_minus_env`, robust intrinsic ratio
  - policy: near-bound ratio, action entropy, sign-flip rate, action-equality diagnostics
  - regime: bull/bear/sideways split deltas

If outcome and mechanism conflict, prioritize mechanism diagnosis before proposing new methods.

### 4.3 Required falsification and significance checks
For any claim beyond smoke:
- at least one mechanism falsification test (e.g., intrinsic postprocess swap, candidate replacement, regime split)
- seeds >= 3 (prefer >= 5)
- report mean/std + bootstrap CI
- for pairwise claims (`Gk` vs `G0`), report paired significance test

Smoke runs:
- allowed for plumbing only
- must be labeled `smoke-only, not for conclusions` in `decisions.md`

### 4.5 Mechanism-first discipline (anti score-chasing)
- Do not optimize for positive gain alone.
- Every optimization attempt must answer two questions in `analysis.md`:
  1) `Mechanism changed?` (what causal path was altered)
  2) `Why this should work in trading?` (market micro/portfolio risk rationale)
- If gain improves but mechanism evidence is missing, the result is exploratory-only.

### 4.4 Leakage control (must document)
- Split boundaries for train/val/test per window
- Execution timestamp rule (decision at `close_t`, fill at `open_t1` unless changed)
- No future information in revise/intrinsic calculations

## 6) Analysis Template (anti logic-break)

Each `analysis.md` round section must follow this exact structure:
1) **Problem** (what failed or was ambiguous)
2) **Mechanism hypothesis** (why it may happen)
3) **Intervention** (what changed; config/code/artifact)
4) **Falsification result** (did evidence support/reject hypothesis)
5) **Residual uncertainty** (what remains unknown and next test)
6) **Mechanism checkpoint** (explicitly answer: `what mechanism changed`, `why this should work in trading`)

No section should skip from result to conclusion without mechanism evidence.

### 6.1 Writing style for analysis/archive docs (mandatory)
- Goal: make the record understandable to humans first, not only concise for machines.
- Required style:
  - use plain language and concrete examples when describing mechanism and failure mode;
  - keep **medium detail**: not one-line bullets, not long essays;
  - explicitly explain key metric meanings when first used (e.g., `action_equal_ratio`, `reward_total_minus_env`);
  - separate facts, interpretation, and decision in different paragraphs.
- Anti-patterns (do not use):
  - over-compressed "keyword-only" conclusions that hide causal logic;
  - very long text without structure or without direct evidence links;
  - only reporting "gain up/down" without mechanism explanation.
- Suggested paragraph order for each key finding:
  1) what happened (observation),
  2) why it likely happened (mechanism),
  3) what we changed or will change (action),
  4) what evidence would falsify this explanation (next check).

- Expand seeds (narrow CI)
- Run run-level CI + falsification suite
- Update paper via `writing/main_vX.tex` (new version only)

## 9) What to do when uncertain
When you are not sure:
- Prefer adding a falsification test over adding complexity
- Prefer documenting assumptions over hiding them in code
- Prefer smaller, auditable changes over large refactors
